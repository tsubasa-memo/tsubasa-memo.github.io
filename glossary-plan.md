# 用語集セクション設計書（全66本）

## コンセプト
ツバサが仕事中に出くわした用語を調べたメモ。素人が素人のために書く体裁。
画像・カメラ用語は東京レタッチのGSCデータで表示回数の大きいキーワードを狙い撃ち。
EC業務用語はツバサのペルソナ（メーカーEC担当）が遭遇する用語を網羅。

---

## ディレクトリ構造

glossary/ 配下にindex.html + 個別用語ページ66本。
slug命名は英語。日本語slugは使わない。

### 特別な扱い
- **ささげ**: 既に sasage-outsource.html（通常記事）が存在するため、用語集には個別ページを作らない。glossary/index.html の用語一覧に「ささげ → 詳しくはこちら」として sasage-outsource.html へリンクする。

## 個別用語ページのテンプレート

### スペック
- 文字数: 1,000〜4,000文字（内容に必要な分量で書く。引き延ばさない、端折らない）
- h2: 3個（固定構成。ただし「〜との違い」を含む記事は4個）
- FAQ構造化データ: 5問
- 参照URL: 本文中に5つ以上
- 日付表示: なし
- SVGサムネイル: なし
- hero色: 用語集共通で統一（記事ごとに変えない）
- 著者プロフィール: 既存のものをそのまま使う
- Google Analytics + AdSense: 通常記事と同じ

### 固定構成

```
リード文（100〜150文字）
→ 仕事で○○という言葉に遭遇した。意味がわからなかったので調べてみた。

h2: ○○とは
  → 定義を2〜3文。参照元URLを本文中に自然に組み込む。

h2: EC実務 / 撮影で出てくる場面
  → ツバサが遭遇した具体的な場面を2〜3個。参照元URL。

h2: 覚えておきたいポイント
  → 実務上の注意点。既存記事・他の用語集への内部リンク。参照元URL。

（※ 「〜との違い」クエリがある記事のみ）
h2: ○○と△△の違い → 比較表で整理。

FAQ構造化データ（5問）
関連する用語（他の用語集ページへの相互リンク）
関連する記事（通常記事へのリンク）
著者プロフィール（既存のものをそのまま使う）
```

## 執筆ルール

### 参照ルール（必須）
- 1記事あたり5つ以上の外部URLを参照元として本文中に記載する
- 参照URLは本文中に自然にリンクとして組み込む（「調べたらこういう記事を見つけた」「○○の公式ヘルプにはこう書いてあった」）
- 外部リンクは `target="_blank" rel="noopener"` を付ける
- 商業サービスサイトは `nofollow` を追加

### 参照元の種類（優先順位）
1. 公式ドキュメント（Google Developers、Apple、Adobe、各ECモール、Shopify等）
2. 技術メディア（ics.media、Web担当者Forum等）
3. 老舗印刷会社のオウンドメディア（大日本印刷、凸版印刷、竹尾等。信頼性が高い）
4. 比較・検証記事（実測データがあるもの）
5. 用語解説記事（定義の確認用）

### 参照元の制限ルール
- **レタッチ会社のコラムは東京レタッチのみ引用可**。他のレタッチ会社・写真修正サービス会社のコラムは引用禁止
- **新興のWeb制作会社・SEO会社のコラムは慎重に扱う**。AIで適当に書かれている記事が多いため、実測データや具体的な検証がない記事は参照元にしない。記事内に独自の検証・スクリーンショット・具体的な数値がある場合のみ引用可
- **老舗の印刷会社のオウンドメディアは積極的に引用してよい**

### 文体・トーン
- 既存ルール通り「だ・である」調
- 「調べてみた」「まとめてみた」のメモ口調
- 専門家ぶらない。わからなかったことを正直に書く

### WebP・AVIFの遭遇場面に関する補足
WebPもAVIFも「誰かから納品される」形式ではなく、Web配信時に変換するフォーマット。デザイナーがAVIFやWebPで制作データを渡すことはない。EC担当が遭遇するのは：PageSpeed Insightsで警告される / CDNが自動変換 / 競合サイトの画像保存時 / Web制作会社からの提案

### 参照チェック
- [ ] 本文中の外部参照URLが5つ以上あるか
- [ ] 東京レタッチ以外のレタッチ会社・写真修正サービスのコラムを引用していないか
- [ ] 新興Web制作会社の記事を引用している場合、独自の検証・実測データがある記事か
